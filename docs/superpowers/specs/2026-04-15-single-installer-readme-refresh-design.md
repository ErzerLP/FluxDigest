# FluxDigest 单入口安装器与 README 首页重构设计

- 日期：2026-04-15
- 主题：统一为单一公开安装入口，使用交互式菜单降低部署门槛，并将 README 首页重构为 Welcome / Feature / Quick Start 结构
- 设计状态：已完成交互式方案确认，待进入实现计划
- 工作区：`D:\Works\guaidongxi\RSS`
- 适用平台：Linux（Docker 部署）

## 1. 背景

当前仓库虽然已经具备完整的 Docker 全栈安装能力与历史的 systemd 部署能力，但对首次用户来说，入口依然不够统一：

- 公开可见的部署脚本不止一个，用户需要判断该执行哪个脚本。
- README 首页承担了过多说明职责，既像手册又像变更日志，不像标准项目首页。
- 用户希望看到的是“进入仓库 -> 执行一个脚本 -> 跟随菜单完成部署 -> 拿到访问地址与默认账号密码”。

本轮设计不再讨论多种并行入口，而是明确把仓库收敛到一个主入口，并围绕它重写 README 首页。

## 2. 已确认约束

以下约束已经在对话中确认，本设计直接作为硬约束采纳：

1. **只保留 1 个公开入口脚本**。
2. 公开入口固定为仓库根目录：`install.sh`。
3. 脚本默认启动后进入**交互式菜单**。
4. 所有需要用户“选择”的场景，都必须采用**方向键上下选择 + 回车确认**的交互方式。
5. 安装模型继续采用 **Docker / Docker Compose**。
6. 平台统一为 **Linux**，不再细分 Ubuntu 22.04 / 24.04 等首页口径。
7. 脚本对系统依赖执行**自动检测**，但**不自动安装**；缺失时只报错并提示用户自行安装。
8. README 首页结构固定为：`Welcome`、`Feature`、`Quick Start`。
9. README 首页风格要靠近标准开源项目首页，不再使用“这是干嘛的”一类口语标题。

## 3. 候选方案与最终选择

### 方案 A：Docker-only 单入口安装器（选中）

在仓库根目录新增 `install.sh`，作为唯一公开入口。该脚本负责：

- 拉起交互菜单
- 统一组织安装 / 自定义安装 / 升级 / 回滚 / 查看部署信息
- 调用内部 Docker 安装逻辑完成真正部署

**优点：**

- 用户只需要记住一个脚本
- 与“全部使用 Docker”的既有决策完全一致
- 文档、演示、排障路径可全部围绕一个入口收敛
- 后续维护成本最低

### 方案 B：单入口包装器，内部继续多脚本并列

表面只暴露一个脚本，但 README 仍同时强调多个历史脚本入口。

**不选原因：**

- 只是表面统一，用户心智仍会分裂
- 长期会重新退化成“多个部署路径共存”的状态

### 方案 C：仅保留首次安装入口，升级 / 回滚只写文档

**不选原因：**

- 生命周期不完整
- 不符合已存在的升级 / 回滚需求

### 结论

采用 **方案 A：Docker-only 单入口安装器**。

## 4. 总体设计目标

本轮最终交付的设计目标如下：

1. 用户进入仓库后，第一条主路径只有：
   ```bash
   bash install.sh
   ```
2. 安装器启动后进入 TUI 交互菜单，使用方向键与回车完成所有选择操作。
3. 用户可以在同一入口中完成：
   - 快速安装
   - 自定义安装
   - 升级现有部署
   - 回滚历史版本
   - 查看当前部署信息
4. README 首页只保留 Welcome / Feature / Quick Start 三段主结构，并把详细说明下沉到 `docs/`。

## 5. 安装器架构设计

## 5.1 公开入口与内部实现分层

### 公开入口

- `install.sh`（仓库根目录）

这是唯一对外推荐的安装入口。

### 内部实现层

保留现有脚本作为内部模块，而不是用户直接入口：

- `deploy/stack/install.sh`
- `deploy/stack/scripts/*.sh`
- 现有 `deploy/scripts/*.sh` 历史脚本

边界定义如下：

- 根目录 `install.sh`：负责交互、模式选择、参数整理、调用内部流程。
- `deploy/stack/install.sh`：负责 Docker 栈渲染、凭据生成、启动、bootstrap、健康检查。
- `deploy/stack/scripts/*.sh`：负责模板渲染、公共函数、Miniflux/Halo bootstrap 等模块化能力。
- `deploy/scripts/*.sh`：保留为兼容 / 高级路径，不再作为 README 主入口。

这样既满足“对用户只有一个脚本”，又避免一次性推倒已有内部实现。

## 5.2 交互技术选型

安装器 TUI 采用 **`whiptail`**。

选择它的原因：

- 在 Linux 终端中足够常见、稳定
- 天然支持方向键导航与确认按钮
- 支持菜单、确认框、输入框、密码框、多选框
- 维护成本明显低于手写 ANSI 菜单

### 交互规则

- **选择类场景**：全部使用 `whiptail` 菜单 / 单选 / 多选
- **输入类场景**：全部使用 `whiptail` 输入框 / 密码框

因此用户视角会是统一的：

- 选择时：方向键上下移动，回车确认
- 输入时：弹窗输入，不出现凌乱的命令行 prompt

## 6. 菜单流设计

## 6.1 主菜单

脚本启动后显示主菜单：

1. 快速安装（推荐）
2. 自定义安装
3. 升级现有部署
4. 回滚到历史版本
5. 查看当前部署信息
6. 退出

这是整个安装器唯一的一级入口。

## 6.2 快速安装流程

目标：面向小白，把步骤压缩到最低。

建议流程：

1. 选择部署组合：
   - FluxDigest + Miniflux + Halo
   - FluxDigest + Miniflux
   - FluxDigest + Halo
   - 仅 FluxDigest
2. 输入基础信息：
   - 安装目录
   - 宿主机访问地址（IP / 域名）
3. 检测是否存在代理环境变量，并允许用户确认是否复用
4. 展示安装摘要：
   - 将部署的组件
   - 将使用的端口
   - 将生成的随机凭据
   - 将写入的目录
5. 用户确认后执行安装
6. 安装完成后展示结果页与摘要文件位置

### 快速安装原则

- 绝大多数字段使用推荐默认值
- 自动生成随机密码
- 自动复用可安全复用的已有配置
- 尽量避免暴露复杂技术细节给首次用户

## 6.3 自定义安装流程

目标：保留灵活性，但不牺牲交互式体验。

用户可逐步选择：

1. 部署组合
2. 是否自定义端口
3. 是否自定义数据库 / 管理员凭据
4. 是否初始化 Miniflux / Halo
5. 是否写入代理相关环境变量
6. 是否立即启动并执行 bootstrap

### 设计原则

- 依然不要求用户手改脚本
- 通过多步菜单完成高级配置
- 保留推荐默认值，降低误操作概率

## 6.4 升级流程

从主菜单进入“升级现有部署”后执行：

1. 自动检测已有安装目录与现有 `.env`
2. 读取当前部署信息
3. 展示升级摘要：
   - 保留现有数据目录
   - 保留现有账号密码 / token / 密钥
   - 重新构建或更新当前代码版本
4. 用户确认后执行升级
5. 执行健康检查
6. 展示升级结果与失败时的回滚提示

### 升级原则

- 绝不默认覆盖已有密钥和业务配置
- 以保留现有状态为第一原则
- 失败时给出明确回滚入口

## 6.5 回滚流程

从主菜单进入“回滚到历史版本”后执行：

1. 检测可用的历史 release / 版本记录
2. 使用菜单列出候选项
3. 用户方向键选择目标版本
4. 展示回滚提示
5. 确认后执行回滚
6. 执行健康检查
7. 展示回滚结果

### 回滚原则

- 不要求用户手工输入 release 目录
- 所有选择动作都在菜单内完成
- 结果页明确告诉用户当前回到哪个版本

## 6.6 查看当前部署信息

该菜单项用于展示：

- 当前部署 profile
- FluxDigest / Miniflux / Halo 访问地址
- 数据目录与 Compose 目录
- 配置文件路径
- 安装摘要文件路径
- 当前管理员用户名

安全要求：

- 默认不在普通状态页全文回显敏感密码
- 密码与 token 统一提示查看受限摘要文件

## 7. 依赖检测与错误处理

## 7.1 依赖检测

根入口脚本在进入正式菜单前检测以下依赖：

- `bash`
- `docker`
- `docker compose`
- `curl`
- `git`
- `whiptail`

若缺失，则直接弹出清晰错误信息并退出，不自动安装。

## 7.2 错误处理原则

任何失败场景都应输出“对用户可理解”的结果，而不是把原始 Bash 日志当主信息。错误提示至少应包含：

- 失败步骤
- 可能原因
- 建议查看的日志或文件位置
- 安装目录 / 摘要文件位置

典型失败场景包括：

- Docker 不可用
- `whiptail` 缺失
- 端口冲突
- 镜像拉取失败
- 某服务健康检查失败
- bootstrap 失败

## 8. 安装完成输出设计

安装成功后，脚本需要统一生成并展示以下信息：

- FluxDigest 访问地址
- Miniflux 访问地址（若部署）
- Halo 访问地址（若部署）
- FluxDigest 默认管理员用户名 / 密码
- Miniflux 默认管理员用户名 / 密码（若部署）
- Halo 默认管理员用户名 / 密码（若部署）
- PostgreSQL 用户名 / 数据库名 / 端口
- Redis 端口
- 配置文件路径
- 安装摘要文件路径
- 下一步建议：
  1. 登录 FluxDigest WebUI
  2. 配置 LLM
  3. 跳转 Miniflux 添加 RSS
  4. 配置 Halo 发布通道

该摘要既要在界面展示，也要落盘保存，便于后续查看。

## 9. README 重构设计

## 9.1 首页主结构

README 首页采用以下主结构：

1. `Welcome`
2. `Feature`
3. `Quick Start`

这是首页主干，其他内容只能作为短附录或文档导航，不得重新把首页写成长手册。

## 9.2 Welcome

Welcome 负责回答三个问题：

1. FluxDigest 是什么
2. 它处理什么输入
3. 最终输出什么结果

推荐包含：

- 一句话项目定位
- 一张轻量流程图（如 Mermaid）
- 三行核心价值概述

不再使用“这是干嘛的”这样的口语标题。

## 9.3 Feature

Feature 采用标准开源项目式特性列表，分组说明：

- Core Workflow
- Publishing & Output
- Admin & WebUI
- Deployment

目标是让用户扫一眼就能判断项目能力完整度，而不是在首页阅读冗长段落。

## 9.4 Quick Start

Quick Start 只保留一条绝对主路径：

```bash
git clone https://github.com/ErzerLP/FluxDigest.git
cd FluxDigest
bash install.sh
```

随后说明：

- 脚本会启动交互菜单
- 可选择 Full / FluxDigest + Miniflux / FluxDigest + Halo / FluxDigest only
- 安装完成后会输出所有服务入口、管理员账号密码、数据库信息与配置文件路径
- 首次使用时应先登录 FluxDigest，配置 LLM，再进入 Miniflux 管理 RSS

## 9.5 文档边界

README 首页不再承载以下内容：

- 部署细节全文
- 升级 / 回滚实现细节
- 长篇接口说明
- 大段目录结构解释
- 故障排查全文

这些内容统一下沉到 `docs/`。

建议保留一个简短的 `Documentation` 导航区，链接到：

- 部署详解
- 集成配置
- Open API 文档
- 其他后续手册

## 10. 改动范围

## 10.1 本轮核心改动文件

- 新增：`install.sh`
- 重构：`README.md`
- 调整：`deploy/stack/install.sh`（作为内部实现层）
- 必要时补充：部署说明文档与安装器参考文档

## 10.2 本轮明确不做

- 不重构业务处理逻辑
- 不重做 WebUI 功能
- 不改 OpenAPI 契约
- 不新增 Kubernetes / Helm / Ansible 方案
- 不把所有历史脚本立即删除到只剩一个文件

说明：本轮的“只有一个脚本”指的是**对用户公开推荐的入口只有一个**，而不是仓库内部不能存在辅助脚本。

## 11. 验证要求

实现阶段至少需要验证：

1. `bash install.sh` 能正确进入交互菜单
2. 所有选择类步骤都能通过方向键 / 回车完成
3. 快速安装路径可以完成一次完整部署
4. 升级与回滚入口能正确触发既有流程
5. README 首页能让首次用户只看三段结构就理解项目与安装路径
6. 详细说明已正确下沉到 `docs/`

## 12. 设计结论

本轮不再继续扩展多个并行安装入口，而是明确收敛为：

- 一个对外入口：根目录 `install.sh`
- 一个主交互模型：`whiptail` 菜单
- 一个首页结构：Welcome / Feature / Quick Start

这样既满足小白一键部署体验，也能在不推倒现有内部实现的前提下完成结构收敛。
